# Repository Guidelines

## Project Structure & Module Organization
- `src/zeliard.cpp` is the current C++ entry point and SDL startup code.
- `src/town/`, `src/mdt/`, and `src/grp/` hold the current native port code for town scenes, MDT parsing, and GRP decoding/unpacking.
- `tools/` contains Python utilities for inspecting and extracting game data; the main viewers are `tools/mdtviewer/`, `tools/grpviewer/`, `tools/spriteeditor/`, and `tools/sfxripper/`.
- `docs/` contains the repo-local analysis notes that describe the current reverse-engineering state, including town, MDT, GRP, collision, runtime, audio, and dungeon notes.
- `asm/` stores original assembly sources, helper scripts, and comparison material from the DOS version.
- `game/` contains extracted original game data and DOS binaries, including the mirrored `game/0/` resource set. Treat it as reference data unless a task explicitly targets it.
- `assets/` contains converted music and sound assets. Treat it as reference data unless a task explicitly targets it.
- `out/` is generated by CMake and should not be edited by hand.

## Build, Test, and Development Commands
- `cmake --preset x64-debug` configures a Windows debug build in `out/build/x64-debug`.
- `cmake --build --preset x64-debug` builds the executable and copies SDL3 next to it.
- `out\build\x64-debug\Zeliard.exe` runs the current build.
- `cmake --preset x64-release` creates a Windows release build.
- Additional configure presets exist for `x86-debug`, `x86-release`, `linux-debug`, and `macos-debug`, but the checked-in build preset currently targets `x64-debug` only.
- There is no dedicated automated test suite yet; use build + launch as the normal validation path.

## Coding Style & Naming Conventions
- Follow the existing C++ style: 4-space indentation, braces on their own lines, and compact control flow.
- Prefer clear, project-owned names over abbreviations. Use PascalCase for constants and constant-like identifiers, such as `WindowWidth`, `InternalHeight`, and `WindowTitle`.
- Do not rename SDL, CMake, or other external API symbols.
- Python tools generally follow standard 4-space indentation and readable snake_case helper names.

## Working Preferences
- Keep changes narrowly scoped to the requested task. Do not add unrelated refactors, renames, or extra variants.
- Treat `game/` and `assets/` as reference data unless the user explicitly asks for changes there.
- Keep generated build output in `out/` untouched unless a build step requires it.
- When the task touches reverse-engineering or data-extraction code, inspect the repo-local evidence first and prefer existing tools, scripts, and docs over outside references.
- For town, MDT, GRP, collision, audio, and runtime work, prefer the matching analysis notes in `docs/` before editing code.
- If a result is still uncertain, say so plainly and label it as a guess instead of presenting it as confirmed.
- Use comments only when they explain a useful reason, workaround, fragile behavior, or performance tradeoff.

## Testing & Validation
- Validate C++ changes by rebuilding the relevant preset and launching the app.
- For tool changes, run the affected Python script against a representative file or folder and confirm the output looks correct.
- For analysis-only work, keep it read-only and report the evidence you inspected instead of changing assets or generated output.
- Keep generated files and build artifacts out of commits.

## Reporting
- Keep final responses focused on changed files or assets, important renamed identifiers, validation results, and any remaining issues.
- Avoid long narrative summaries when a short implementation report is enough.

## Commit & Pull Request Guidelines
- The current history in this checkout is sparse, so keep commits short, imperative, and descriptive, for example: `Add SDL startup window`.
- Pull requests should summarize the change, mention any data or asset impact, and include screenshots or logs when behavior changes are visible.
- Keep changes narrowly scoped to the requested task.
